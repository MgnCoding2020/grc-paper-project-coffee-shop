# Log Review Procedure  
Burn and Churn Coffee

---

## 1. Purpose
The purpose of this procedure is to define the process for reviewing system and security logs to identify suspicious activity, potential security incidents, and operational issues.

Regular log review supports early detection of unauthorized access, system misuse, and other security events.

---

## 2. Scope
This procedure applies to logs generated by systems used in Burn and Churn Coffee operations, including:

- Google Workspace login and account activity logs  
- Toast POS transaction and administrative logs  
- Network and router activity logs (if available)  
- Incident tracking records  

---

## 3. Roles and Responsibilities

### 3.1 Business Owner / Management
- Ensures log review activities are performed  
- Reviews escalated security events  
- Approves response actions for significant findings  

### 3.2 System Administrator
- Performs log reviews  
- Identifies suspicious or unusual activity  
- Documents findings and escalates issues  

### 3.3 Supervisors / Managers
- Assist in validating business-related activity  
- Support investigation of suspicious events  

---

## 4. Log Review Frequency

Logs should be reviewed on a **weekly basis**.

Additional reviews may occur:
- After security incidents  
- When suspicious activity is reported  
- Following system changes or updates  

---

## 5. Log Review Process

### Step 1: Collect Logs
The system administrator gathers relevant logs from applicable systems.

Examples include:
- Google Workspace login activity  
- POS transaction logs  
- Incident records  

---

### Step 2: Review Activity
Logs should be reviewed for:

- Unusual login locations or times  
- Repeated failed login attempts  
- Unexpected administrative activity  
- Suspicious POS transactions  
- Any activity inconsistent with normal operations  

---

### Step 3: Identify Suspicious Events
If suspicious activity is identified:

- Document the event  
- Determine whether it may indicate a security incident  
- Escalate to management if necessary  

---

### Step 4: Escalation
If activity indicates a potential incident:

- Initiate the appropriate incident response playbook  
- Continue monitoring related activity  

---

### Step 5: Documentation
Log review results must be documented in:

- artifacts/monitoring/incident-log.md  

Documentation should include:
- Date of review  
- Systems reviewed  
- Findings  
- Actions taken  

---

## 6. Tools and Supporting Evidence

Log review may be supported by:

- Python validation scripts (incident_log_review_check.py)  
- System dashboards and administrative consoles  
- Incident log documentation  

---

## 7. Record Retention
Log review documentation should be retained for at least one year to support internal review and auditing.

---

## 8. Procedure Review
This procedure should be reviewed annually or when significant changes occur to systems or monitoring capabilities.
